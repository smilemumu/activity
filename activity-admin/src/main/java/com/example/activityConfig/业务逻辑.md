# 活动配置后台 — 业务逻辑文档

## 1. 系统概述

**活动配置后台管理系统**（Activity Admin）是一个营销活动配置管理平台，面向消费金融领域，用于创建和管理注册类营销活动。系统支持页面注册和弹窗注册两种活动形式，提供活动生命周期管理、页面组件配置、奖励管理、规则配置、分享配置及数据报表等功能。

### 业务背景

- **归属公司**：xxx公司（消费金融）
- **归属机构**：机构A（101）、机构B（301）
- **涉及产品**：产品A（0302）、产品B（PLON）、产品C（0303）、产品D（0305）、产品E（0307）、产品F（SFCL）

## 3. 核心业务流程

### 3.1 活动生命周期

```
创建(草稿) ──→ 发布 ──→ 下线
   │            │         │
   DRAFT    PUBLISHED   CLOSED
```

- **创建**：新建活动保存为 DRAFT 状态，同时保存归属机构和归属部门关联
- **发布/下线**：通过修改状态字段变更活动生命周期阶段
- **编辑**：更新活动信息时，机构关联和部门关联采用"先删后插"策略

### 3.2 活动配置关系

每个活动（ACTIVITY）作为核心实体，关联以下子配置：

```
                    ┌──────────────────┐
                    │     ACTIVITY     │
                    │  (活动主表)       │
                    └──────┬───────────┘
          ┌────────┬───────┼───────────────┬──────────┐
          ▼        ▼       ▼               ▼          ▼
    活动-机构   活动-部门   页面配置      规则配置    奖励配置
      关联        关联    (PAGE_CONFIG)  (RULE_CONFIG) (REWARD_CONFIG)
    (ORG_REL)  (DEPT_REL)      │              │           │
                               │         ┌────┴───┐       │
                               │         ▼        ▼       │
                          排序展示   活动-产品  参与条件  分享配置
                          (前端用)  关联(PROD  (CONDITION) (SHARE_CONFIG)
                                    _REL)
```

### 3.3 用户认证流程

```
客户端请求 POST /api/auth/login {username, password}
    │
    ▼
AuthService.login()
    ├─ 查询 SYS_USER 表 → 用户不存在 → 返回 400 "用户名或密码错误"
    ├─ 检查 STATUS='Y' → 被禁用 → 返回 400 "账号已被禁用"
    ├─ BCrypt 密码比对 → 不匹配 → 返回 400 "用户名或密码错误"
    └─ 生成 JWT Token（24小时有效），返回 LoginResponse
```

- **默认管理员**：admin / admin123
- **Token 存储方式**：客户端保存在 localStorage，请求时附加在 `Authorization: Bearer <token>` 头
- **无状态会话**：服务端不保存 session，每次请求通过 JWT 过滤器验证 token

## 4. API 接口文档

### 4.1 基础约定

- **Base URL**：`http://localhost:8080`
- **Content-Type**：`application/json`
- **认证方式**：`Authorization: Bearer <token>`（除登录和 OPTIONS 外全部需要）
- **统一响应格式**：

```json
{
  "code": 200,       // 200=成功, 400=业务错误, 500=系统错误
  "message": "...", // 描述信息
  "data": {}         // 响应数据
}
```

- **是否需要认证**：否
- **请求体**：

```json
{
  "username": "admin",
  "password": "admin123"
```


- **是否需要认证**：是
- **响应**：返回当前登录用户的 `SysUser` 对象（密码字段已置空）

### 4.3 活动管理模块 (ActivityController)

#### POST `/api/activity` — 创建活动

- **请求体**：

```json
{
  "activityType": "PAGE_REGISTER",
  "title": "新人注册送好礼",
  "description": "活动描述",
  "startTime": "2026-06-01T00:00:00",
  "endTime": "2026-06-30T23:59:59",
  "coverImage": "/uploads/xxx.png",
  "orgRels": [{"orgCode": "101", "orgName": "机构A"}],
  "deptRels": [{"deptName": "消费者权益保护部", "contactName": "夏生超"}]
}
```

- **校验规则**：`activityType` 和 `title` 不能为空；`startTime` 和 `endTime` 不能为空
- **业务逻辑**：
  1. 将 ActivityRequest 转为 Activity 实体，状态默认为 DRAFT
  2. 设置 createdBy 为当前登录用户名
  3. 插入活动主表（MyBatis 自动回填 ID）
  4. 批量插入机构关联关系
  5. 批量插入部门关联关系
  6. 整个过程在一个事务内完成
- **响应**：返回新创建的活动 ID

#### PUT `/api/activity/{id}` — 更新活动

- **业务逻辑**：
  1. 校验活动是否存在（不存在抛 BusinessException）
  2. 更新活动主表
  3. 删除旧的机构关联 → 插入新的机构关联
  4. 删除旧的部门关联 → 插入新的部门关联
  5. 整个过程在一个事务内完成

#### GET `/api/activity/{id}` — 获取活动详情

- **响应**：返回 ActivityResponse，包含活动信息、机构关联列表、部门关联列表

#### GET `/api/activity/page` — 分页查询活动列表

- **请求参数**：
  - `keyword`：按标题模糊搜索（可选）
  - `status`：按状态筛选（可选）
  - `page`：页码，从 1 开始
  - `size`：每页条数，默认 10
- **排序**：按创建时间倒序
- **响应**：`PageResult<ActivityResponse>`，包含 rows、total、page、size

#### PUT `/api/activity/{id}/status` — 更新活动状态

- **请求参数**：`status` — DRAFT / PUBLISHED / CLOSED
- **业务逻辑**：用于活动发布和下线操作

```json
  "code": 200,
  "data": {
    "ruleConfig": { ... },    // 规则配置对象

  }
}
```

#### PUT `/api/activity/{activityId}/rule-config` — 保存规则配置

- **请求体**：

```json
{
  "ruleConfig": {
    "hasReward": "Y",
    "rewardDeliveryType": "AUTO",
    "ruleType": "REGISTER",
    "totalClaimLimitEnabled": "Y",
    "totalClaimLimit": 1000,
    "regionLimitEnabled": "N",
    "mutualExclusiveEnabled": "N"
  },
  "products": [
    {"productCode": "0302", "productName": "产品A"}
  ],
  "conditions": [
    {"conditionType": "NEW_USER", "productFilter": "{\"include\":[\"0302\"]}"}
  ]

```

- **业务逻辑**：
  1. 校验活动是否存在
  2. 规则配置采用 upsert 策略：存在则更新，不存在则插入
  3. 产品关联和条件配置采用"先删后插"策略
  4. 整个过程在一个事务内完成

- **规则字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| hasReward | CHAR(1) | 是否有奖励，Y/N |
| rewardDeliveryType | VARCHAR(20) | 奖励发放方式（如 AUTO） |
| ruleType | VARCHAR(30) | 规则类型，默认 REGISTER |
| totalClaimLimitEnabled | CHAR(1) | 是否开启总领取上限 |
| totalClaimLimit | INT | 总领取上限数 |
| regionLimitEnabled | CHAR(1) | 是否开启地区限制 |
| mutualExclusiveEnabled | CHAR(1) | 是否开启互斥 |
| mutualExclusiveActivityId | BIGINT | 互斥活动 ID |

## 5. 数据库设计

### 5.1 表概览（11 张表）

| 表名 | 说明 | 与 ACTIVITY 关系 |
|------|------|------------------|
| ACTIVITY | 活动主表 | — |
| ACTIVITY_ORG_REL | 活动-机构关联 | 多对一 |
| ACTIVITY_DEPT_REL | 活动-部门关联 | 多对一 |
| PAGE_CONFIG | 页面组件配置 | 多对一 |
| RULE_CONFIG | 规则基础配置 | 一对一（UNIQUE） |
| ACTIVITY_PRODUCT_REL | 活动-产品关联 | 多对一 |
| PARTICIPATION_CONDITION | 参与条件配置 | 多对一 |
| REWARD_CONFIG | 奖励配置 | 多对一 |
| SHARE_CONFIG | 分享配置 | 一对一（UNIQUE） |
| SYS_USER | 系统用户 | 独立 |
| REWARD_CLAIM_RECORD | 奖励领取记录 | 多对一 |

### 5.2 核心表详细说明

#### ACTIVITY（活动主表）

| 字段 | 类型 | 说明 |
|------|------|------|
| ID | BIGINT | 主键，自增 |
| ACTIVITY_TYPE | VARCHAR(30) | 活动类型：PAGE_REGISTER / POPUP_REGISTER |
| TITLE | VARCHAR(100) | 活动标题 |
| DESCRIPTION | VARCHAR(500) | 活动描述 |
| START_TIME | DATETIME | 开始时间 |
| END_TIME | DATETIME | 结束时间 |
| COVER_IMAGE | VARCHAR(500) | 封面图片 URL |
| STATUS | VARCHAR(20) | 状态：DRAFT / PUBLISHED / CLOSED，默认 DRAFT |
| CREATED_BY | VARCHAR(50) | 创建人 |
| CREATED_TIME | DATETIME | 创建时间，自动填充 |
| UPDATED_TIME | DATETIME | 更新时间，自动更新 |

#### RULE_CONFIG（规则配置）

每个活动仅一条规则配置记录（ACTIVITY_ID 为 UNIQUE 约束），控制活动的核心规则：

- **是否有奖励 (hasReward)**：控制是否展示奖励模块
- **奖励发放方式 (rewardDeliveryType)**：定义奖励发放机制
- **规则类型 (ruleType)**：默认 REGISTER（注册类活动）
- **总领取上限 (totalClaimLimit)**：活动整体领取次数限制
- **地区限制 (regionLimitEnabled)**：是否限制参与地区（功能开关，详细规则可扩展）
- **互斥活动 (mutualExclusiveEnabled + mutualExclusiveActivityId)**：与另一个活动互斥，用户不能同时参与

#### REWARD_CONFIG（奖励配置）

支持五种奖励类型：

| 类型编码 | 中文名 | 说明 |
|---------|--------|------|
| POINTS | 积分 | 发放平台积分 |
| COUPON | 优惠券 | 发放优惠券 |
| EXCHANGE_CODE | 兑换码 | 发放兑换码（需对接兑换系统） |
| THANKS | 谢谢参与 | 无奖励，用于抽奖场景 |
| MALL_COUPON | 商城商品券 | 商城专属商品券 |

关键字段：
- **stock**：库存数量，可控制奖励发放总量
- **maxClaimTimes**：单用户最大领取次数
- **occupyWinCount**：是否计入中奖次数（用于控制中奖概率）
- **conditionJson**：JSON 格式的额外发放条件（灵活扩展）

#### PAGE_CONFIG（页面配置）

用于配置活动页面的可视化组件：
- **configType**：组件类型（BANNER、IMAGE、FORM、BUTTON 等）
- **configValue**：LONGTEXT 类型，可存储 HTML 片段或 JSON 配置
- **width/height**：组件尺寸
- **sortOrder**：展示排序（保存时自动按数组顺序分配）

#### PARTICIPATION_CONDITION（参与条件）

- **conditionType**：条件类型（如 NEW_USER 新用户、EXISTING_USER 老用户等）
- **productFilter**：LONGTEXT 格式的 JSON，定义产品过滤规则（白名单/黑名单）

## 6. 安全与权限

### 6.1 认证机制

```
请求 → JwtAuthenticationFilter → 提取 Authorization 头
    ├─ 无 Token 或 Token 无效 → 匿名访问（被 SecurityConfig 拦截，返回 401/403）
    └─ Token 有效 → 解析用户名 → 设置 SecurityContext → 继续请求链
```

- **签名算法**：HS512（HMAC-SHA512）
- **Token 有效期**：24 小时（86400000ms）
- **Token 载荷**：userId、username、sub（subject=username）、iat（签发时间）、exp（过期时间）

### 6.2 权限控制

- `POST /api/auth/login`：允许匿名访问
- `OPTIONS` 请求：允许匿名访问（CORS 预检）
- 其他所有接口：需要有效 JWT Token
- CSRF：已禁用（适用于 RESTful API）
- Session：无状态模式（STATELESS）

### 6.3 密码安全

- 采用 BCrypt 加密存储
- 默认强度：BCryptPasswordEncoder 默认 10 轮 hash
- 初始化管理员密码 hash：`$2b$10$.BJl4x3scKSKHdaSUI3IluQ04P6z9fH1S.KJSFJY6.2UWSrMjFGg6` 对应明文 `admin123`

## 8. 异常处理

### 8.1 异常体系

```
Exception
├── BusinessException (RuntimeException)
│   └── code: 错误码（默认 400）
│   └── message: 错误描述
└── 其他 RuntimeException / Exception
    └── 统一按 500 返回
```

### 8.2 GlobalExceptionHandler 处理策略

| 异常类型 | HTTP 状态码 | 响应 code | 处理方式 |
|---------|-----------|-----------|---------|
| BusinessException | 200 OK | e.getCode() | 返回业务错误码和信息 |
| BindException | 400 Bad Request | 400 | 返回校验失败的第一条消息 |
| Exception | 500 Internal Server Error | 500 | 记录日志，返回"服务器内部错误" |

> **注意**：BusinessException 返回 HTTP 200，实际错误信息通过响应体中的 code 字段区分。这样设计是为了前端能统一处理响应而不需要在 HTTP 层面区分。

### 8.3 常见业务异常场景

| 场景 | code | message |
|------|------|--------|
| 用户名或密码错误 | 400 | 用户名或密码错误 |
| 账号被禁用 | 400 | 账号已被禁用 |
| 活动不存在 | 400 | 活动不存在 |
| 奖励配置不存在 | 400 | 奖励配置不存在 |
| 文件上传为空 | 500 | 上传文件为空 |
| 文件写入失败 | 500 | 文件上传失败: ... |

## 10. 数据流转说明

### 10.1 创建活动的完整数据流

```
前端 (Vue 3 + Ant Design)
  │  用户填写活动信息、选择机构、选择部门
  │
  ▼
POST /api/activity
  │  ActivityRequest: { activityType, title, ..., orgRels, deptRels }
  ▼
ActivityService.create()
  │  1. toEntity → Activity (status=DRAFT)
  │  2. activityMapper.insert() → 主键自增回填
  │  3. saveOrgRels() → 逐个 insert ACTIVITY_ORG_REL
  │  4. saveDeptRels() → 逐个 insert ACTIVITY_DEPT_REL
  │  @Transactional 保证原子性
  ▼
返回 activityId → 前端跳转至活动详情页继续配置规则、页面、奖励等
```

### 10.2 规则配置保存流程

```
PUT /api/activity/{id}/rule-config
  │  { ruleConfig: {...}, products: [...], conditions: [...] }
  ▼
RuleConfigService.save()
  │  1. activityMapper.findById() 校验活动存在
  │  2. ruleConfigMapper.findByActivityId() → 有则 update，无则 insert
  │  3. productRelMapper.deleteByActivityId() → 逐个 insert 新产品
  │  4. conditionMapper.deleteByActivityId() → 逐个 insert 新条件
  │  @Transactional 保证一致性
```

### 10.3 Token 认证流程

```
每次请求 → JwtAuthenticationFilter.doFilterInternal()
  │  resolveToken(): 从 Authorization 头取 Bearer token
  │  validateToken(): JWT 签名校验 + 过期校验
  │  SecurityContextHolder.setAuthentication(username)
  ▼
Controller 中通过 SecurityContextHolder.getAuthentication().getName() 获取当前用户
```
